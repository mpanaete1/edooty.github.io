Manage users and policies

  • Configure the HTPasswd identity provider for authentication

    Authentication and Authorization, V4.6, Chapter 5.1.1 : https://access.redhat.com/documentation/en-us/openshift_container_platform/4.6/html/authentication_and_authorization/configuring-identity-providers#configuring-htpasswd-identity-provider

     

    1. create a password file

      yum provides htpasswd

      yum install httpd-tools

      htpasswd -c -B (bcrypt) -b <name of file/htpasswd> <username> <password>

      first time with -c to create the htpasswd file, remove -c to add users

      create additional users: htpasswd -b <file name> <username> <password>

    2. Build a secret (contains the data (base64 encoded))

    oc create secret generic <name/htpasswd-secret> --from-file htpasswd=<location of file created in step 1> -n openshift-config

    1. Custom resource for OAuth

      oc get oauth cluster -o yaml > /tmp/<oauth.yaml>

      Edit to look like the example below.

      copy from documentation <name.yaml>

      xxxxxxxxxx
      apiVersion: config.openshift.io/v1
      kind: OAuth
      metadata:
          name:cluster
      spec:
          identityProvders:
          - name: <htpasswd file name>
            mappingMethod: claim
            type: HTPasswd
            htpasswd:
              fileData:
                name: htpasswd-secret  (name from step 2 <name/htpasswd-secret>)

      Update secret file with new yaml : oc replace -f /location/of/.yaml file

    4. oc get pods -n openshift-authentication

     

  • Create and delete users, Modify user passwords

    Authentication and Authorization, V4.6: Chapter 5.1.7: https://access.redhat.com/documentation/en-us/openshift_container_platform/4.6/html/authentication_and_authorization/configuring-identity-providers#identity-provider-htpasswd-update-users_configuring-htpasswd-identity-provider

extracts the file with the users: oc extract secret/htpasswd-secret -n openshift-config --to <directory> --confirm

Retrieve the htpasswd secret and put in new file:

oc get secret <secret name> -ojsonpath={.data.htpasswd} -n openshift-config | base64 --decode > newhtpasswd

added a user to htpasswd - htpasswd -b newhtpasswd <user> <pass>

Replace the secret object with updated users in newhtpasswd

oc create secret generic <secret name> --from-file htpasswd=newhtpasswd -n openshift-config --dry-run=client -o yaml | oc replace -f -

remove user - htpasswd -D htpasswd <user>

Update the secret : oc set data secret/htpasswd-secret --from-file htpasswd=<new directory>

set service account: oc set serviceaccount deployment/<name> <service account name>

Control access to resources

  • Define role-based access controls, Apply permissions to users

    Authentication and Authorization, v.4.6, Chapter6.5-6-* :https://access.redhat.com/documentation/en-us/openshift_container_platform/4.6/html/authentication_and_authorization/using-rbac

     

    Local Role - only manage elements at a project level

    oc adm policy add-cluster-role-to-user/group <cluster role> <username>

    when you see adm its a cluster role

    Remove - oc adm policy remove-cluster-role-to-user/group <cluster role> <username>

    Check access - oc adm policy who-can delete user

    Add specified role to a user: oc policy add-role-to-user <role name> <username> -n <project name>

     

  • Create and apply secrets to manage sensitive information

    create generic secret with key-value pair

    oc create secret generic <secret name> --from-literal key1=secret1 --from-literal key2=secret2

    create from files

    oc create secret generic ssh-keys --from-file id_rsa=/path-to/id_rsa --from-file id_rsa.pub=/path-to/id_rsa.pub

    Create a TLS secret

    oc create secret tls secret-tls --cert /path-to-certificate --key /path-to-key

    Expose secrets to pod : oc create secret generic demo-secret --from-literal user=demo-use --from-literal root_password=zT1KTgk

    Secret as Pod Env Variable: oc set env deployment/demo --from secret/demo-secret --prefix MYSQL_

Mount a file system inside a pod: oc set volume deployment/demo --add --type secret --secret-name demo-secret --mount-path /app-secrets

 

  • Create service accounts and apply permissions using security context constraints

    Authentication and Authorization, V4.6, Chapter 13.4 :https://access.redhat.com/documentation/en-us/openshift_container_platform/4.6/html/authentication_and_authorization/managing-pod-security-policies#security-context-constraints-about_configuring-internal-oauth

    1. openssl genrsa -out <file.key> 2048

    2. openssl req -new -x509 -key privkey.pem -out selfcert.pem

    3. openssl x509 -n <cert.pem> -noout -text

  • Secure routes using TLS certificates

    Networking, V4.6, Chapter 14.2: https://access.redhat.com/documentation/en-us/openshift_container_platform/4.6/html/networking/configuring-routes#configuring-default-certificate

    Error: failed to open tls.cert

    oc create secret tls <secret-name> --cert <cert.crt> --key <key.key>

    Create TLS secret using .pem keys

    oc create secret tls <name-tls> --cert <name-combined.pem> --key <name-key.pem>

    mount the secret to the right directory described in logs

    oc set volumes deployment/<name> --add --type secret --secret-name <secret-name> --mount-path <directory needed for crt>

     

    Use the openssl x509 command with the -noout and -ext 'subjectAltName' to read the newly created .pem cert.

    openssl x509 -in <namepem-combined.pem> --noout -ext 'subjectAltName'

    DNS: <name>.wildcard.domain.name>

     

    Create a passthrough routes to the service pointing to the domain.

    oc create route passthrough --service <service-name> --port <port> --hostname <<name>./wildcard/domain/FQDN for route>

Adding resource quota and limitrange

xxxxxxxxxx
- apiVersion: v1
  kind: LimitRange       (LIMIT RANGE)
  metadata:
    name: project-limitrange
  spec:
    limits:
    - default:
        memory: 100Mi
        cpu: 100m
      defaultRequest:
        memory: 30Mi
        cpu: 30m
      type: Container
- apiVersion: v1
  kind: ResourceQuota    (Resource Quota)

    name: project-quota

      limits.memory: 4Gi

Configure cluster scaling

  • Manually control the number of cluster workers

    oc scale --replicas <# of pods> deployment/<app name>

  • Automatically scale the number of cluster workers

    oc autoscale deployment/<name> --min 1 --max 5 --cpu-percent 80
